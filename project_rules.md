后端开发规范文档（TypeScript + NestJS + AWS EB + RDS）
一、项目基本信息
- 业务范围：美国地区官网 + 支付系统 + 管理后台（含数据看盘、代理商管理、自动邮件）
- 并发要求：支持三位数并发
- 订单量级：日均两位数订单
- 部署环境：AWS Elastic Beanstalk（EB）
- 数据库：AWS RDS MySQL 8.0（独立实例，不与 EB 环境绑定，避免环境删除导致数据丢失）
二、技术栈规范（强制遵循）
- 开发语言：TypeScript（强类型，禁止使用 any 类型，避免隐性 Bug）
- 开发框架：NestJS（最新稳定版，企业级规范架构，适配后台管理+支付业务）
- ORM 框架：TypeORM（与 NestJS 适配性最优，统一数据库操作）
- 邮件服务：Nodemailer（支持 SMTP / AWS SES 对接，实现自动邮件发送，支持附件、多语言、移动端适配）
- 身份验证：JWT（登录鉴权、接口权限控制）
- API 风格：RESTful（统一接口规范，便于前后端对接）

四、核心业务模块详细需求
4.1 数据看盘（Dashboard）
- 核心统计数据（必含）：今日销售额统计、冷静期单量、待激活订单数量、退款申请数量、已完成订单数量、代理商总数统计
- 补充统计数据：总用户数、总订单数、总交易额、代理商相关统计（代理商总数、各级代理数量、代理总佣金、代理业绩排行）
- 订单状态统计：待支付、冷静期、待激活、已完成、已取消、退款订单的数量及占比
- 趋势统计：近7天/30天的订单量、交易额、佣金金额、退款数量趋势（接口返回 JSON 格式，支持前端绘图）
- 数据要求：实时统计，无需 websocket 实时推送，刷新页面即可获取最新数据
- 权限控制：仅超级管理员、经销商可查看全量/名下代理商看盘数据（含所有统计项，含代理商总数）；代理商仅可查看自身相关看盘数据（仅含自身关联的今日销售额、自身冷静期单量、自身待激活/退款申请/已完成订单数量，不含代理商总数及其他代理商数据）
4.2 代理商管理（Agent）
- 核心功能：实现无限级分销关系管理，支持任意层级代理商关联，分销关系可通过树状结构可视化展示
- 代理商基础信息：id、账号（登录名）、用户名、密码、邀请码（关键渠道标识）、姓名、联系电话、上级代理id（parent_id）、代理等级、佣金比例、账号状态（启用/禁用）、创建时间、订单总数、销售总金额、总佣金金额
- 权限控制（严格遵循）：
        
  - 超级管理员：可查看所有代理商数据（全量）、操作所有代理商相关功能
  - 经销商：可查看自身名下所有层级代理商数据，可操作自身名下所有代理商相关功能，不可操作非自身名下代理商
  - 代理商：仅可查看自身数据（基础信息、自身订单、自身佣金），不可查看任何下级代理商数据及相关统计，无任何操作权限（仅查看）
- 列表展示（必含字段）：代理商 ID、邀请码、所属上级（父级代理商，显示父级代理商ID+用户名）、订单总数、创建时间、销售总金额、总佣金、佣金金额
- 代理商关联订单：每个代理商可查看自身关联的所有订单列表，订单列表除原有字段外，需额外增加“每单佣金（美金）”字段；支持订单列表导出功能（导出格式支持Excel，包含所有订单字段）
- 树状结构展示：可视化展示代理商无限级分销关系树，支持展开/折叠查看下级代理，显示每级代理商的订单贡献（订单总数、销售总金额、总佣金）
- 经销商管理功能（仅经销商可操作自身名下代理商）：
        
  - 添加新代理：需填写登录名、用户名、邀请码、密码（自动加密存储），可指定上级代理（仅可指定自身名下的代理商作为上级）
  - 指定上级代理：新增/编辑代理商时，可指定上级代理，完善分销关系
  - 修改代理信息：可编辑代理商的用户名、联系电话、佣金比例、账号状态（启用/禁用）
  - 禁用/启用代理账号：可操作自身名下代理商账号的启用/禁用状态，禁用后代理商无法登录系统
  - 发放佣金金额：可手动修改代理商的佣金金额（支持调整总佣金、单佣金），修改后同步更新佣金记录
- 超级管理员额外功能：可操作所有代理商，不受层级限制；可批量禁用/启用代理商账号；可调整任意代理商的上级代理及分销关系
- 佣金管理：订单支付完成后，自动计算对应层级代理的佣金，生成佣金记录；经销商手动修改佣金后，同步更新佣金记录并记录操作日志
- 佣金记录：包含订单id、用户id、代理id、每单佣金（美金）、总佣金、计算时间、佣金状态（未结算/已结算）、操作人（手动修改时）
4.3 自动邮件系统（Mail）
- 技术依赖：基于 Nodemailer 开发，支持 SMTP 协议或 AWS SES 对接；支持邮件附件、多语言切换、移动端适配
- 邮件模板要求：品牌化设计，符合整体视觉风格；支持英文（主）、日文、韩文切换；适配移动端显示；包含品牌安全认证标识
- 触发场景及详细要求（必选，共3类邮件）：
        
  1. 邮件1：支付成功通知
            
    - 触发时机：用户支付成功即刻发送
    - 邮件主题：《会员订单确认函》
    - 附件内容：《会员预售协议》PDF
    - 包含信息：用户姓名、证件号、购买日期、邀请码、订单号
  2. 邮件2：冷静期结束通知
            
    - 触发时机：用户支付后第7天自动发送
    - 邮件主题：《会员服务启动通知 》
    - 通知内容：告知用户冷静期结束，即将开通正式权益
    - 补充内容：说明权益开通的预计时间（激活中周期：7-14天）
  3. 邮件3：权益激活通知
            
    - 触发时机：管理员手动点击“确认激活”按钮后发送
    - 邮件主题：《会员激活成功通知》
    - 核心信息：明确列出权益生效日期至到期日期
    - 补充内容：告知用户如何使用会员权益（操作指引）
- 日志记录：所有邮件发送记录（发送时间、接收人、邮件类型、发送状态、语言版本）存入数据库，便于排查问题；超级管理员可查看所有邮件日志，经销商仅可查看自身及名下代理商相关邮件日志，代理商仅可查看自身相关邮件日志
4.4 订单 + 支付模块（基础核心）
- 订单流程（严格遵循）：1. 待支付 → 2. 支付成功 → 3. 冷静期（7天，冷静期内用户可主动申请退款） → 4. 激活中（支付后7-14天，冷静期结束自动切换） → 5. 已完成（管理员手动点击“确认激活”按钮后切换）；冷静期内申请退款则切换为“退款中”，审核完成后切换为“已退款”
- 订单列表展示（必含字段）：订单号（系统自动生成，规则统一）、用户姓名、证件号（完全脱敏）、邮箱（部分脱敏：zhang*@gmail.com）、手机号（完全脱敏）、邀请码（关键渠道标识）、支付方式、交易金额、此单佣金金额、交易 ID（第三方支付流水号）、订单状态、创建时间
- 数据规范：金额字段统一使用 decimal(18,2)（如交易金额、此单佣金金额），避免浮点精度问题；订单号自动生成，确保唯一，格式可配置（如前缀+时间戳+随机码）
- 关联逻辑：订单创建时自动绑定对应代理商（通过邀请码关联），用于后续佣金计算；订单状态流转时，同步更新对应代理商的订单总数、销售总金额、佣金金额
- 支付要求：支持支付接口对接（如 Stripe/PayPal），实现支付回调验签、金额核对，防止篡改；支付成功后即刻触发“支付成功通知”邮件
- 联动逻辑：支付完成后，自动触发佣金计算、对应代理商佣金记录生成、“支付成功通知”邮件发送；冷静期结束后自动触发“冷静期结束通知”邮件；管理员确认激活后，自动触发“权益激活通知”邮件
- 权限控制：超级管理员可查看所有订单数据、操作所有订单相关功能（激活、退款审核）；经销商可查看自身名下代理商的所有订单数据，可操作自身名下代理商关联订单的激活操作；代理商仅可查看自身推广的订单数据，无任何操作权限；用户可在前台查看自身订单，可在冷静期内申请退款
4.5 激活与退款操作（核心流程）
- 冷静期规则：支付成功后进入冷静期，周期为7天；冷静期内，用户可在前台页面主动点击申请退款，后台同步生成退款申请记录
- 退款操作：用户申请退款后，后台显示“退款申请数量”（同步至数据看板）；管理员可查看退款申请，审核通过后，订单状态切换为“已退款”，同步触发退款回调至支付平台，更新佣金状态（已退款订单佣金作废）
- 激活流程：
        
  - 冷静期结束（支付后第7天），订单自动从“冷静期”切换为“激活中”状态，同时触发“冷静期结束通知”邮件
  - 激活中周期：7-14天（从支付成功之日起计算，即第7天至第14天）
  - 确认激活：管理员在后台查看“待激活订单数量”（同步至数据看板），对激活中订单手动点击“确认激活”按钮，订单状态切换为“已完成”，同时触发“权益激活通知”邮件
- 状态管控：订单状态流转需记录操作日志（操作人、操作时间、状态变更原因）；已完成订单不可再申请退款；已退款订单不可再进行激活操作
4.6 其他基础模块
- Auth 模块：JWT 登录鉴权，支持超级管理员、经销商、代理商、普通用户不同角色权限区分，登录时自动识别角色并分配对应权限
- Admin 模块：超级管理员对全量数据的查看、编辑、操作权限（订单、用户、代理商、邮件日志等）；经销商仅可操作自身及名下代理商相关数据；代理商仅可操作自身相关数据
- User 模块：官网用户注册、登录、个人信息管理；用户可查看自身订单，可在冷静期内申请退款
- Product 模块：商品 CRUD、分类管理、库存管理、上下架操作；超级管理员、经销商可操作全量商品数据；代理商仅可查看商品数据，无操作权限
- Permission 模块（新增）：维护角色（超级管理员、经销商、代理商）与权限的关联关系，实现权限的精细化管控；支持权限动态分配，可新增角色权限配置；重点管控代理商管理、订单激活/退款、数据看板查看等核心权限
五、数据库规范（AWS RDS MySQL 8.0）
5.1 通用字段（所有表必须包含）
id BIGINT PRIMARY KEY AUTO_INCREMENT COMMENT '主键ID',
created_at DATETIME DEFAULT CURRENT_TIMESTAMP COMMENT '创建时间',
updated_at DATETIME NULL ON UPDATE CURRENT_TIMESTAMP COMMENT '更新时间',
is_deleted TINYINT DEFAULT 0 COMMENT '软删除标识（0=未删除，1=已删除）'
5.2 字段规范
- 金额相关字段：统一使用 DECIMAL(18,2)（如交易金额、此单佣金金额、总佣金金额），禁止使用 float/double；佣金金额需区分“每单佣金（美金）”和“总佣金”
- 状态字段：
        
  - 订单状态（order_status）：TINYINT 枚举，注释：0=待支付，1=冷静期，2=激活中，3=已完成，4=退款中，5=已退款，6=已取消
  - 角色字段（role）：TINYINT 枚举，注释：0=超级管理员，1=经销商，2=代理商，3=普通用户
  - 代理商状态（agent_status）：TINYINT 枚举，注释：0=禁用，1=启用
  - 佣金状态（commission_status）：TINYINT 枚举，注释：0=未结算，1=已结算，2=已作废（退款订单）
- 脱敏字段：证件号（完全脱敏，存储时加密，返回时显示全星号）、手机号（完全脱敏，返回时显示全星号）、邮箱（部分脱敏，返回时隐藏@前除首字符外的内容，如zhang*@gmail.com）
- 索引规范：普通索引命名为 idx_表名_字段（如 idx_order_agent_id、idx_agent_parent_id），唯一索引命名为 uk_表名_字段（如 uk_agent_account、uk_agent_invite_code、uk_order_order_no）；新增 idx_order_status（订单状态索引）、idx_agent_status（代理商状态索引）、idx_commission_status（佣金状态索引）
- 关联规范：禁止使用数据库外键，通过业务层逻辑实现表与表之间的关联（便于后续扩展、迁移）；代理商表 parent_id 字段关联上级经销商/代理商ID（超级管理员无上级，parent_id 为 null）；订单表 agent_id 字段关联对应代理商ID，invite_code 字段关联代理商邀请码；订单表增加 refund_apply_time（退款申请时间）、activate_time（激活时间）字段，记录对应操作时间
5.3 核心表清单（必含）
- users（普通用户表，含用户姓名、证件号、邮箱、手机号等）
- admins（管理员表，含超级管理员、经销商，关联 role 字段区分角色）
- agents（代理商表，关联 parent_id 字段，含登录名、用户名、邀请码、佣金比例等所有代理商基础信息）
- agent_commissions（代理商佣金记录表，含每单佣金、总佣金、佣金状态等）
- products（商品表）
- categories（商品分类表）
- orders（订单表，含所有订单列表字段，关联 agent_id、invite_code，新增退款、激活相关字段）
- order_items（订单项表）
- payments（支付记录表，含交易ID、支付方式、支付金额等）
- refund_records（退款记录表，含退款申请时间、审核状态、审核人等）
- mail_logs（邮件发送日志表，含邮件类型、语言版本、附件信息等）
- roles（角色表，存储超级管理员、经销商、代理商角色信息）
- permissions（权限表，存储各类操作权限）
- role_permissions（角色权限关联表，绑定角色与对应权限）
- operation_logs（操作日志表，记录订单状态变更、代理商信息修改、佣金调整等操作）
六、API 规范
6.1 接口路径规范
- 基础路径：/api/v1/资源名称（复数形式）
- 核心接口路径示例：
       
  - 数据看盘：/api/v1/dashboard（统计数据）
  - 订单管理：/api/v1/orders（订单列表、激活、退款）、/api/v1/orders/export（订单导出）
  - 代理商管理：/api/v1/agents（列表、新增、编辑）、/api/v1/agents/tree（树状结构）、/api/v1/agents/commission（佣金修改）
  - 邮件管理：/api/v1/mail/logs（邮件日志）
  - 退款管理：/api/v1/refunds（退款申请、审核）
- 权限区分：接口需携带 JWT token，后端通过 token 解析角色，根据角色返回对应权限的数据；无权限操作时，直接返回403错误
6.2 请求方法规范
- GET：查询数据（单条/列表/统计、树状结构、邮件日志）
- POST：新增数据（新增代理商、退款申请）
- PUT：全量更新数据（修改代理商信息）
- PATCH：部分更新数据（修改代理商佣金、订单激活、退款审核）
- DELETE：删除数据（软删除，修改 is_deleted 为 1）
- GET：/api/v1/orders/export（订单导出接口，返回Excel文件流）
6.3 统一返回格式
{
  "code": 200,
  "message": "success",
  "data": {} // 单个对象/数组/统计数据，无数据时返回空对象；无权限时 data 为空，message 提示无操作权限
}
6.4 错误码规范
- 200：请求成功
- 400：参数错误（如必填项缺失、格式错误）
- 401：未登录（JWT 失效/未携带 token）
- 403：无权限（当前角色无法操作该接口或查看该数据）
- 404：资源不存在（如订单id不存在、代理商id不存在）
- 409：业务冲突（如冷静期已过无法申请退款、已完成订单无法退款）
- 500：服务器异常（如数据库报错、接口报错、邮件发送失败）
6.5 分页规范
- 分页参数：page（页码，默认1）、size（每页条数，默认10）
- 分页返回数据：
{
  "code": 200,
  "message": "success",
  "data": {
    "list": [], // 分页数据列表（根据角色权限返回对应数据）
    "page": 1, // 当前页码
    "size": 10, // 每页条数
    "total": 100 // 总条数（仅显示当前角色有权限查看的总条数）
  }
}
七、安全规范（支付业务重点）
- 密码安全：用户/管理员/代理商密码使用 bcrypt 加密存储，禁止明文存储；新增代理商时，密码自动加密后存入数据库
- 敏感字段：返回数据时，严格按要求脱敏（证件号、手机号完全脱敏，邮箱部分脱敏）；数据库中敏感字段（证件号、手机号）需加密存储
- 接口防护：所有接口添加防重复提交机制（如通过 token 或请求签名）；权限校验前置，无权限用户直接返回403错误，不执行后续业务逻辑；订单激活、退款审核、佣金修改等核心操作需校验操作人权限
- 支付安全：支付回调必须验签，防止伪造回调；金额核对，确保支付金额与订单金额一致；退款操作需校验订单状态，防止非法退款
- 权限控制：每个接口添加权限守卫，严格区分超级管理员、经销商、代理商的操作权限，防止越权操作（如代理商无法调用经销商的新增代理商接口、无法查看下级数据）
- 注入防护：通过 TypeORM 参数绑定，防止 SQL 注入；接口入参校验，防止 XSS 攻击；邮件内容需过滤非法字符，防止邮件注入
- 日志安全：操作日志需完整记录核心操作（订单状态变更、代理商信息修改、佣金调整、退款审核），包含操作人、角色、操作时间、操作内容，便于追溯
八、部署适配规范（AWS Elastic Beanstalk）
- 无状态服务：项目设计为无状态，避免依赖本地文件、内存存储（适配 EB 自动扩缩容）
- 配置读取：所有配置（数据库地址、JWT 密钥、邮件配置、PDF 附件路径等）从 EB 环境变量读取，避免硬编码
- 文件存储：禁止本地文件读写，商品图片、附件（《会员预售协议》PDF）等文件存储到 AWS S3，邮件发送时从 S3 读取附件
- 扩容适配：支持多实例部署，接口无状态，可随并发量自动扩缩容；订单状态流转、邮件发送等核心逻辑需支持分布式部署，避免并发问题
- 数据库连接：RDS 连接使用内网地址，提高连接速度和安全性，避免公网暴露；配置数据库连接池，适配三位数并发需求
- 定时任务：冷静期结束自动切换订单状态、自动发送邮件等定时任务，需适配 EB 部署环境，确保定时执行稳定
九、代码规范
- 命名规范：
        
  - 文件名：kebab-case（如 user.controller.ts、permission.guard.ts、order.service.ts）
  - 类名：PascalCase（如 UserController、OrderService、PermissionGuard、AgentService）
  - 方法名/变量名：camelCase（如 getUserById、orderAmount、checkAgentPermission、calculateCommission）
  - 数据库表名：下划线复数（如 users、agent_commissions、role_permissions、refund_records）
  - 数据库字段名：下划线（如 user_id、commission_amount、parent_id、role_id、order_no、refund_apply_time）
  - 枚举名：PascalCase，枚举值：UPPER_CASE（如 OrderStatus、RoleType）
- 注释规范：每个类、方法必须添加 JSDoc 注释，说明功能、参数、返回值、异常情况；权限相关方法需注明可操作角色；核心业务逻辑（订单状态流转、佣金计算、邮件触发）需添加详细注释，便于维护
- 逻辑规范：每个函数只做一件事，避免函数过于庞大；复杂业务逻辑拆分到 service 层，controller 层仅负责接收请求、返回响应；权限校验逻辑统一封装到 PermissionGuard 中，全局复用；订单状态流转、佣金计算、邮件发送等核心逻辑单独封装，便于复用和修改
- 异常处理：所有异常统一通过全局异常过滤器捕获，返回统一格式的错误信息，禁止直接抛出原生错误；无权限操作时，抛出403异常，提示明确的无权限原因；业务异常（如冷静期已过无法退款）需抛出409异常，提示清晰的业务限制
- 日志规范：统一使用 NestJS 内置 logger，按 info（普通日志）、warn（警告日志）、error（错误日志）分级记录，便于问题排查；越权操作、核心业务操作（激活、退款、佣金修改）需记录日志（含操作人、角色、操作接口、时间、操作内容）；邮件发送失败需记录错误日志，便于排查问题
- 多语言规范：邮件模板按语言分类（英文、日文、韩文），通过配置文件控制语言切换；多语言文本统一管理，避免硬编码
